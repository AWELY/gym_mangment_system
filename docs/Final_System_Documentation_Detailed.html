1. Software Requirements Specification (SRS)

1.1 Purpose

This document provides a comprehensive and detailed specification, architecture design, and static analysis report for the Glory Gym Management System. The system is built as a Windows desktop application (WinForms, C#) designed to manage all operations of a modern gym, including memberships, subscriptions, trainers, point-of-sale (POS) store, and diet plans.

1.2 User Classes and Roles

  • System Administrator (Admin): Has unrestricted access to all modules. Can view high-level financial reports, manage trainers, configure system settings, and add/remove users.
  • Receptionist: Has operational access. Restricted to managing Members, conducting POS transactions (Store), and viewing/sending Diet plans. Cannot view financial reports or trainer salaries.

1.3 Detailed Functional Requirements (FR)

  • FR-AUTH: The system shall provide a secure login screen. Authentication relies on local data validation.
  • FR-MEM: The system shall allow users to Add, Edit, Delete, and Search members. Member data includes personal details and active subscription plans.
  • FR-SUB: The system shall maintain a catalog of Subscription Plans to be assigned to members.
  • FR-POS: The system shall include a POS module where users can add items to a cart, calculate totals, and checkout, automatically deducting from inventory.
  • FR-FIN: The system shall calculate monthly net profit by subtracting trainer salaries from the gross revenue.

1.4 External Integrations (WhatsApp API)

A critical business requirement is direct communication with members. The system supports two modes of WhatsApp integration:

  • Qonvo API (Automated): Used primarily in the Diet module. The system reads QonvoApiToken and QonvoBaseUrl from App.config to automatically dispatch PDF Diet Plans directly to the member's WhatsApp number. Statuses are logged in the DietSendHistory.
  • Browser Fallback (Manual): For general chats, the system opens the default web browser to https://wa.me/{phone} with a pre-filled, URL-encoded Arabic greeting so the Receptionist can quickly message members without saving their numbers to a phone.

2. Software Design Document (SDD)

2.1 Architectural Pattern

The application uses a Monolithic WinForms Architecture. It separates concerns logically into three implicit layers:

  1. Presentation Layer (UI): Contains all .cs Form classes handling user interactions and input validation.
  2. Business Logic Layer (BLL): Contains managers like AppSession, and ThemeManager.
  3. Data Access Layer (DAL): GymDataStore is responsible for serializing/deserializing the JSON objects to the local drive.

2.2 External API Modules

To support the WhatsApp requirements defined in the SRS, the architecture includes dedicated service classes:

  • QonvoWhatsApp.cs: Handles HTTP POST requests to the Qonvo API endpoints, attaching PDF binaries and formatting the JSON payloads required for automated messaging.
  • WhatsAppWeb.cs: A utility class that sanitizes phone numbers and generates the correct URI strings to launch the system's default browser for manual chat initiation.

6. Database Schema & Normalization Analysis

6.1 Schema BEFORE Normalization (Flat JSON structure)

In the original state, the system stores flat data where values are duplicated and non-atomic. This violates basic relational database design principles.

EntityColumns (Flat Structure)Normalization Issues
Members Id, FullName, Phone, Gender, PlanName, PriceText, DurationText, JoinDate Violates 2NF. Plan details (Price, Duration) are duplicated for every member who buys the same plan.
StoreSales Id, SoldAt, TotalAmount, Summary Violates 1NF. The Summary column contains a comma-separated list of items (e.g. "2x Water, 1x Towel"), which is not atomic.

6.2 Normalization Steps

  • First Normal Form (1NF): Ensure atomic values. The comma-separated Summary in StoreSales is extracted into a new SaleItems table.
  • Second Normal Form (2NF): Remove partial dependencies. The Members table previously held the Plan Name, Price, and Duration directly. We removed these and replaced them with a PlanId.
  • Third Normal Form (3NF): Eliminate transitive dependencies. By performing the 2NF split, 3NF is inherently achieved. All non-key attributes strictly depend on the Primary Key.
